# Logging Before Monolog

**Before PSR-3, before Monolog, before structured logging existed as a
concept — MIOLO built its own.**

Today, PHP logging means Monolog (2011) with PSR-3 (2012): you configure
handlers (file, syslog, Slack, Elasticsearch), formatters (JSON, line),
and processors (add context, add stack traces). Every modern framework
ships with Monolog out of the box.

In 2001, there was nothing. If you wanted logging, you wrote it yourself.

---

## What Didn't Exist

- **Monolog** — first release 2011
- **PSR-3 (LoggerInterface)** — finalized 2012
- **Log levels as a standard** (DEBUG, INFO, WARNING, ERROR) — formalized
  by syslog (RFC 5424, 2009) but not adopted in PHP until PSR-3
- **Structured logging** (JSON logs, context arrays) — mainstream ~2015
- **Log aggregation** (ELK stack, Splunk, Datadog) — enterprise 2010s

PHP's built-in logging was `error_log()` — a single function that could
write to a file, send an email, or append to the system logger. That was it.

---

## MIOLO's Custom Logging Stack

MIOLO implemented a complete logging infrastructure with three components:
**MLog** (message logging), **MTrace** (execution tracing), and **MProfile**
(performance profiling).

### MLog: Four-Handler Logging System

**File:** `classes/services/mlog.class.php`

MLog supports four output strategies, selected by configuration:

#### File Handler
```php
private function handlerFile($msg)
{
    $logfile = $this->home . '/' . trim($this->host) . '.log';
    $ts = $this->manager->getSysTime();
    error_log($ts . ': ' . $msg . "\n", 3, $logfile);
}
```

One log file per hostname. Timestamped. Appended with `error_log()` mode 3
(write to specific file).

#### Socket Handler
```php
private function handlerSocket($msg)
{
    if ($this->port) {
        if (!$this->socket) {
            $this->socket = fsockopen($host, $this->port);
        }
        fputs($this->socket, $msg . "\n");
    }
}
```

Streams log messages to a TCP port — for remote monitoring. Open a terminal
with `nc -l -p 9999` and watch logs flow in real-time.

#### Database Handler
```php
private function handlerDb($msg)
{
    $db = $this->manager->getDatabase('miolo');
    $idLog = $db->getNewId('seq_miolo_log', 'miolo_sequence');
    $sql = new sql('idlog, timestamp, msg, host', 'miolo_log');
    $db->execute($sql->insert(array($idLog, $ts, $msg, $this->host)));
}
```

Logs to a database table with auto-incrementing ID, timestamp, message,
and hostname. Queryable. Persistent. This is essentially what Monolog's
`DoctrineHandler` or `PDOHandler` does today.

#### Screen Handler
```php
private function handlerScreen($msg)
{
    $this->content .= "document.addDebugInformation('" .
        str_replace("\n", '', addslashes(nl2br($msg . "\n"))) . "');";
}
```

Injects JavaScript to display log messages in the browser — a precursor
to PHP Debug Bar (2013).

### SQL Logging: The Query Auditor

```php
public function logSQL($sql, $force = false, $conf = '?')
{
    $sql = preg_replace("/\n+ */", " ", $sql);
    $sql = trim($sql);
    $line = "$ip - [$dts] - $conf - $uid \"$sql\"";
    $logfile = $this->home . '/' . $fileName . '-sql.log';
    error_log($line . "\n", 3, $logfile);
}
```

Every SQL query is logged in an Apache-log-like format: IP, timestamp,
database config name, user ID, and the full query. This is the same
information that Laravel Telescope (2018) captures, but written to a
flat file in 2001.

### Comparison: MLog (2001) vs Monolog (2011)

| Feature | MIOLO MLog (2001) | Monolog (2011) |
|---|---|---|
| Handlers | 4 (file, socket, database, screen) | 30+ (file, syslog, Slack, Redis, etc.) |
| Configuration | PHP array in config file | PHP code or YAML/XML |
| Log levels | Numeric (0=off, 1=errors, 2=all) | PSR-3 (DEBUG through EMERGENCY) |
| Formatters | Hardcoded per handler | Pluggable formatter classes |
| Context | IP + timestamp + message | Arbitrary context arrays |
| SQL logging | Built-in with Apache-like format | Separate (Doctrine logger, Telescope) |
| Browser output | JavaScript injection | PHP Debug Bar, Clockwork |
| Interface | Custom methods | PSR-3 LoggerInterface |

## MTrace: Execution Tracing

**File:** `classes/services/mtrace.class.php`

MTrace maintains an in-memory array of everything that happens during a
request — every file included, every SQL query, every handler dispatched.

```php
public function trace($msg, $file = '', $line = 0)
{
    $message = $msg;
    if ($file != '') $message .= " [file: $file] [line: $line]";
    $this->trace[] = $message;
    $this->log->logMessage('[TRACE]' . $message);
}
```

Stack traces are generated by throwing and catching an exception — the
only way to get a backtrace in PHP 4 before `debug_backtrace()` existed:

```php
public function traceStack($file = '', $line = 0)
{
    try { throw new Exception; }
    catch (Exception $e) { $strStack = $e->getTraceAsString(); }
    $this->trace($strStack, $file, $line);
    return $strStack;
}
```

At the end of a request, the entire trace can be rendered as an HTML table
and appended to the page footer — a request lifecycle inspector, like
Laravel Debugbar or Symfony Profiler.

## MProfile: Performance Profiling

**File:** `classes/services/mprofile.class.php`

A hand-rolled profiler using `microtime()` with explicit enter/exit
semantics:

```php
public function profileEnter($name)
{
    $this->profile[$name][0] = $this->profileTime(); // start timestamp
    $this->profile[$name][1] = 0;                    // accumulated time
    $this->profile[$name][2] = true;                 // state: active
}

public function profileExit($name)
{
    $this->profile[$name][1] += $this->profileTime() - $this->profile[$name][0];
    $this->profile[$name][2] = false;
}

public function profileDump()
{
    foreach (array_keys($this->profile) as $name) {
        $usage = sprintf("%.3f", $this->profile[$name][1]);
        $this->log->logMessage("[PROFILE]$name: $usage sec");
    }
}
```

Every database call, every file include, every handler dispatch is wrapped
in `profileEnter` / `profileExit`. The dump shows exactly how long each
phase took — the same information that Blackfire (2014) or Xdebug profiling
provides, but built entirely in userland PHP.

## Global Error Configuration

**File:** `classes/support.inc.php`

Error reporting was configured globally at framework startup:

```php
if (!$MIOLO || !$MIOLO->getConf('options.debug')) {
    error_reporting(E_ALL & ~E_NOTICE & ~E_WARNING & ~E_DEPRECATED);
    ini_set("log_errors", "on");
    ini_set("error_log", "../var/log/php_error.log");
}
```

In debug mode, everything was shown. In production, notices and warnings
were suppressed and errors were written to a file. This binary switch
(debug on/off) is the ancestor of environment-based configuration
(`APP_ENV=production` / `APP_DEBUG=false`).

## What This Tells Us

MIOLO's logging stack implements every concept that modern logging
frameworks provide: multiple output targets, SQL query auditing, request
tracing, performance profiling, browser output. The difference is
standardization — PSR-3 gave PHP a common interface so you could swap
Monolog for any other logger. MIOLO's MLog is tightly coupled to the
framework, but the engineering decisions are identical.

The socket handler is particularly notable: streaming logs to a remote
port in 2001 is conceptually the same as sending logs to a Logstash
collector in 2015. The infrastructure was manual (`netcat` instead of
Elasticsearch), but the architecture is the same.
